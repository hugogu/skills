# Java Review Checklist (AI Assisted)

> **定位说明**：本 checklist 是**参考和灵感来源**，不是必须逐条勾选的评分表。Review 时应先理解代码的业务意图，用工程师常识判断合理性，再回头从这里寻找可能被遗漏的角度。对于命名规范、格式缩进、行长度、import 顺序等纯风格问题，**不单独提出**。

## Business Semantics & Common Sense

- [ ] **业务意图匹配**：代码中的常量、枚举、硬编码值是否与当前业务场景匹配
  - 例如：交易凭证、合同、审计日志等通常需要长期保存，写死 `SEVEN_DAY`、`TEMP`、`7天` 可能不符合业务常识
  - 例如：支付/金额类接口使用 `int` 或 `float` 而非 `BigDecimal` / 分单位整数
- [ ] **枚举值语义**：选择的枚举值是否表达了正确的业务状态，是否存在更合适的枚举值
- [ ] **默认值陷阱**：默认值是否对当前业务过于宽松或过于严格（如超时时间、重试次数、缓存时间、审批阈值）
- [ ] **文案与提示**：用户可见的文案、错误提示是否准确、得体，是否存在歧义或误导
- [ ] **业务规则硬编码**：折扣率、费率、上限、有效期等规则是否应配置化

## Architecture & Design

- [ ] **单一职责**：类/方法是否承担了过多职责，是否存在 God Class / God Method 的迹象
- [ ] **抽象层次**：接口设计是否稳定，实现细节是否泄漏到上层；是否使用了合适的抽象（接口/抽象类）
- [ ] **SOLID 原则**：
  - 开闭原则：新增功能是否需要大量修改旧代码
  - 依赖倒置：高层模块是否依赖低层细节，而非抽象
  - 里氏替换：子类是否破坏了父类约定
- [ ] **设计模式**：是否可以用更合适的模式简化当前实现（如策略模式替代大量 if/else、模板方法消除重复流程）
- [ ] **分层合理性**：Controller / Service / DAO / Domain 之间的边界是否清晰，是否存在越层调用
- [ ] **依赖方向**：领域层是否依赖了基础设施层（如直接依赖具体 ORM 框架类）
- [ ] **配置 vs 硬编码**：业务规则、阈值、URL、超时时间等是否应抽为配置或常量
- [ ] **扩展性**：新增同类业务时，是否只需要新增少量代码，而不用修改既有核心逻辑

## Logic & Correctness

- [ ] **分支完备性**：if/else/switch 是否覆盖了所有业务场景，是否存在隐式 default/else 遗漏
- [ ] **循环与终止**：while/for 是否有明确的退出条件，是否会死循环或提前终止
- [ ] **状态一致性**：异常发生后对象/事务状态是否合理，是否存在半完成修改（如部分字段已改、部分未回滚）
- [ ] **并发与时序**：
  - 多线程场景下是否存在竞态条件（如先查后改的非原子操作）
  - 锁的粒度是否合理，是否存在死锁风险（嵌套锁、锁顺序不一致）
  - 使用了线程不安全的类（如 SimpleDateFormat、HashMap 并发写）
- [ ] **数值与边界**：除零、整数溢出、越界访问、BigDecimal 构造方式（避免 double 构造）、金额计算精度
- [ ] **幂等性与副作用**：重复调用是否安全（重复提交、消息重试、定时任务重叠执行）
- [ ] **条件判断陷阱**：== 与 equals 混用、短路求值误用、类型隐式转换
- [ ] **时区与日期**：跨时区业务是否使用了带时区的日期类型，夏令时/闰秒是否有影响
- [ ] **异常恢复**：捕获异常后是否有合理的降级或重试策略，而不是简单吞掉
- [ ] **资源泄漏**：数据库连接、线程池、文件流是否在异常路径下也能正确释放

## Security (Human Judgment Required)

- [ ] **SQL 注入**：是否存在动态拼接 SQL（包括 ORDER BY、IN 子句）
- [ ] **越权风险**：接口是否校验了数据归属权（如 A 用户能操作 B 用户的数据）
- [ ] **敏感信息**：日志、异常信息、返回给前端的数据中是否泄漏了密码、token、密钥、个人敏感信息
- [ ] **文件操作**：上传/下载路径是否经过校验，存在路径遍历或任意文件读取风险
- [ ] **反序列化**：是否对不可信数据执行反序列化（如 JSON -> 任意类、原生序列化）
- [ ] **依赖安全**：新增的三方依赖是否有已知 CVE（高风险库）

## Data Integrity

- [ ] **事务边界**：业务操作是否在合适的事务粒度内，是否存在长事务或事务遗漏
- [ ] **缓存一致性**：缓存更新/删除策略是否与数据库写入保持一致，是否存在读写不一致窗口
- [ ] **并发写控制**：对同一资源的并发更新是否有乐观锁/悲观锁/分布式锁控制
- [ ] **数据校验**：核心业务数据在入口层、领域层、持久层是否都有必要的校验（非空、范围、格式）

## Maintainability & Extensibility

- [ ] **魔法数字/字符串**：业务含义不明确的裸数字/字符串是否应提取为具名常量
- [ ] **重复逻辑**：相同业务逻辑是否在多处硬编码，应提取为公共方法/工具/领域服务
- [ ] **注释质量**：复杂算法、非常规实现、业务规则是否有足够注释说明“为什么”而非“做什么”
- [ ] **废弃 API**：是否使用了已明确废弃或不推荐的 API（如 Date、Vector、Hashtable 等）
- [ ] **泛型与类型安全**：集合、反射等场景是否丢失了类型信息，导致运行时才暴露类型错误
