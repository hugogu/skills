# Python Review Checklist (AI Assisted)

> **定位说明**：本 checklist 是**参考和灵感来源**，不是必须逐条勾选的评分表。Review 时应先理解代码的业务意图，用工程师常识判断合理性，再回头从这里寻找可能被遗漏的角度。对于缩进、行长度、引号、import 排序、简单命名规范等纯风格问题，**不单独提出**。

## Business Semantics & Common Sense

- [ ] **业务意图匹配**：代码中的常量、枚举、硬编码值是否与当前业务场景匹配
  - 例如：交易凭证、合同、审计日志等通常需要长期保存，写死 `SEVEN_DAY`、`TEMP`、`timedelta(days=7)` 可能不符合业务常识
  - 例如：金额计算是否使用了 `float` 而非 `Decimal` / 分单位整数
- [ ] **枚举值语义**：选择的枚举值是否表达了正确的业务状态，是否存在更合适的枚举值
- [ ] **默认值陷阱**：默认值是否对当前业务过于宽松或过于严格（如超时时间、重试次数、缓存时间、审批阈值）
- [ ] **文案与提示**：用户可见的文案、错误提示是否准确、得体，是否存在歧义或误导
- [ ] **业务规则硬编码**：折扣率、费率、上限、有效期等规则是否应配置化

## Architecture & Design

- [ ] **单一职责**：函数/类/模块是否承担了过多职责，是否存在 God Class / God Function
- [ ] **抽象层次**：公共接口是否稳定，实现细节是否泄漏到调用方；是否使用了合适的抽象（协议/抽象基类/接口）
- [ ] **依赖方向**：领域层是否依赖了基础设施层（如直接依赖具体 ORM、HTTP 客户端、文件系统）
- [ ] **SOLID 原则**：
  - 开闭原则：新增功能是否需要大量修改旧代码
  - 依赖倒置：高层模块是否依赖低层细节
  - 里氏替换：子类是否破坏了基类约定（如抛出更宽泛的异常、改变方法语义）
- [ ] **配置 vs 硬编码**：业务规则、阈值、URL、超时时间等是否应抽为配置（Pydantic Settings、环境变量、YAML）
- [ ] **扩展性**：新增同类业务时是否只需要新增少量代码（如注册新策略、新处理器），而不需要修改既有核心逻辑
- [ ] **类型抽象**：类型注解是否充分表达了领域模型，是否过度使用 `Any` 或丢失了泛型信息

## Logic & Correctness

- [ ] **分支完备性**：if/elif/else 是否覆盖了所有业务场景，match 是否有 exhaustive 处理
- [ ] **循环终止**：while 循环是否有明确的退出条件，递归是否有深度限制
- [ ] **异常链**：重新抛出异常时是否使用 `raise ... from ...` 保留了原始上下文
- [ ] **异常恢复**：捕获异常后是否有合理的降级或重试策略，而不是简单 `pass` 吞掉
- [ ] **EAFP vs LBYL**：在并发或复杂场景下，try/except 与 if-check 的选择是否合理
- [ ] **可变默认参数**：函数签名中是否使用了 `list`、`dict` 等可变对象作为默认值
- [ ] **并发与时序**：
  - 多线程/多进程共享数据是否使用了锁、队列或原子操作
  - 异步代码中是否存在竞态条件（如先查后改的非原子操作）
  - 使用了非线程安全的对象（如全局可变状态）
- [ ] **字典安全**：动态 key 访问是否使用 `.get()` 或 `in` 检查，避免未预期的 KeyError
- [ ] **浮点比较**：是否避免直接用 `==` 比较浮点数，使用 `math.isclose`
- [ ] **时区处理**：是否混用了 naive 和 timezone-aware datetime，跨时区业务是否正确
- [ ] **拷贝语义**：修改嵌套对象时是否意外 mutates 了共享引用（深浅拷贝问题）
- [ ] **资源泄漏**：文件、网络连接、数据库会话、异步客户端是否在异常路径下也能正确关闭
- [ ] **幂等性**：重复调用（如接口重试、定时任务）是否安全

## Security (Human Judgment Required)

- [ ] **SQL 注入**：是否存在动态拼接 SQL（包括 ORDER BY、IN 子句、表名/列名参数化）
- [ ] **命令注入**：`subprocess` 调用是否使用了 `shell=True` 且拼接了用户输入
- [ ] **路径遍历**：文件上传/读取路径是否经过校验，存在目录遍历风险
- [ ] **反序列化**：不信任数据是否使用了 `pickle.load`、`yaml.unsafe_load` 等危险操作
- [ ] **eval / exec**：是否存在执行动态代码的风险入口
- [ ] **敏感信息**：日志、异常信息、返回给客户端的数据中是否泄漏了密钥、token、个人敏感信息
- [ ] **依赖安全**：新增的三方依赖是否有已知 CVE

## Data Integrity

- [ ] **事务边界**：数据库操作是否在合适的事务粒度内，是否存在长事务或事务遗漏
- [ ] **缓存一致性**：缓存更新/删除策略是否与数据库写入保持一致
- [ ] **并发写控制**：对同一资源的并发更新是否有乐观锁/悲观锁/分布式锁控制
- [ ] **数据校验**：核心业务数据在入口层、领域层、持久层是否都有必要的校验

## Maintainability & Extensibility

- [ ] **魔法数字/字符串**：业务含义不明确的裸数字/字符串是否应提取为具名常量或枚举
- [ ] **重复逻辑**：相同业务逻辑是否在多处硬编码，应提取为公共函数/类/领域服务
- [ ] **注释质量**：复杂算法、非常规实现、业务规则是否有足够注释说明“为什么”
- [ ] **Pythonic 设计**：是否合理使用生成器、上下文管理器、@property 等语言特性提升表达力
- [ ] **兼容性风险**：使用了即将废弃的 API（如 `asyncore`、`imp`）或平台专属特性
